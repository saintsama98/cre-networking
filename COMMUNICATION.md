# CRE Workflow Communication Guide

This document explains how CRE workflows communicate with each other using the HTTP module provided by CRE.

## Communication Format

CRE workflows communicate using **HTTP** with the following characteristics:

### 1. HTTP Protocol
- **Method**: POST (for workflow-to-workflow communication)
- **Content-Type**: `application/json`
- **Authentication**: JWT tokens with cryptographic signatures (ECDSA EVM)

### 2. Request Format

When a workflow sends a request to another workflow, it uses CRE's HTTP client capability:

```go
request := &http.Request{
    Method: "POST",
    Url:    "https://gateway.cre.chain.link/workflows/receiver-id",
    Headers: map[string]string{
        "Content-Type": "application/json",
    },
    Body: jsonPayload,
    CacheSettings: &http.CacheSettings{
        Store:  true,
        MaxAge: durationpb.New(5 * time.Minute),
    },
}
```

### 3. Response Format

The receiver workflow returns a JSON response:

```json
{
  "status": "success",
  "received_at": 1234567890,
  "processed": true,
  "response": "Successfully received and processed: message",
  "echoed_data": {
    "key": "value",
    "processed_by": "receiver_workflow"
  }
}
```

## Key Components

### HTTP Trigger (Receiver)
- Uses `http.Trigger()` from `github.com/smartcontractkit/cre-sdk-go/capabilities/networking/http`
- Accepts incoming HTTP requests
- Validates JWT signatures against `authorizedKeys`
- Passes payload to workflow handler

### HTTP Client (Sender)
- Uses `http.SendRequest()` from CRE SDK
- Sends HTTP POST requests to other workflows
- Handles authentication automatically
- Supports caching to prevent duplicate executions

### Cache Settings
- **Purpose**: Prevent duplicate executions across DON nodes
- **Configuration**: `CacheSettings.Store = true` with `MaxAge` duration
- **Benefit**: Ensures single execution even in distributed environment

## Authentication

### JWT Tokens
- Automatically generated by CRE tools
- Signed with private key (ECDSA EVM)
- Validated against `authorizedKeys` in workflow configuration

### Authorized Keys
Configure in `cre.yaml`:
```yaml
triggers:
  - type: "http"
    config:
      authorizedKeys:
        - type: "ECDSA_EVM"
          publicKey: "0xYourEVMAddress"
```

## Example Communication

### Sender Workflow Input
```json
{
  "message": "Hello from sender",
  "target_url": "https://gateway.cre.chain.link/workflows/receiver-id",
  "data": {
    "key": "value"
  }
}
```

### Receiver Workflow Receives
```json
{
  "message": "Hello from sender",
  "sender_id": "sender_workflow",
  "timestamp": 1234567890,
  "data": {
    "key": "value"
  }
}
```

### Receiver Workflow Responds
```json
{
  "status": "success",
  "received_at": 1234567890,
  "processed": true,
  "response": "Successfully received and processed: Hello from sender",
  "echoed_data": {
    "key": "value",
    "processed_by": "receiver_workflow",
    "original_message": "Hello from sender"
  }
}
```

## Testing Communication

### Local Testing
1. Use CRE test runtime: `testutils.NewTestRuntime()`
2. Test workflows directly without deployment
3. Verify JSON serialization/deserialization

### Integration Testing
1. Deploy workflows to CRE
2. Get receiver workflow URL
3. Configure sender with receiver URL
4. Trigger sender workflow via HTTP
5. Verify end-to-end communication

### Using CRE CLI
```bash
# Simulate receiver workflow
cre workflow simulate \
  --workflow receiver-workflow \
  --input '{"message":"test","sender_id":"test"}'

# Simulate sender workflow
cre workflow simulate \
  --workflow sender-workflow \
  --input '{"message":"test","target_url":"..."}'
```

## Best Practices

1. **Always use CacheSettings** for HTTP requests to prevent duplicate executions
2. **Validate inputs** before processing
3. **Handle errors gracefully** with proper error messages
4. **Use structured logging** via `runtime.Logger()`
5. **Set appropriate timeouts** in cre.yaml
6. **Never expose private keys** in code or logs
7. **Use environment variables** for configuration
8. **Test locally** before deploying
